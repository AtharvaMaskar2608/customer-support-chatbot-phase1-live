1Start here

Executive summary

Choice Jini is a customer-support chatbot for FinX, Choice India's retail trading platform. It ships as a React page inside the FinX Android/iOS app (a WebView) and as a floating window on the FinX website. Its single job: take routine work off the human support team.

Who it's for

Logged-in FinX retail trading clients. The widget is post-login and inherits the app's authentication — that is the core design premise, not a detail.

The problem it kills

"Verification required from your registered email" appears in every support-ticket cluster (58 P&L + 14 Holding + 5 Ledger + 9 Contract-Note tickets). Because Jini is post-login, that entire friction category disappears — reports arrive straight in the chat.

What Jini does — two capabilities

📊 Fetch reports

Eleven deterministic report flows (P&L, Ledger, Tax, Contract Notes, CML and more). The client picks a few options via tappable chips; Jini fetches the file server-side and drops it in the chat as a download card — or emails it to their registered address.

💬 Answer questions

A retrieval-augmented (RAG) function answers explanatory and process questions ("what are my brokerage charges?", "how do I check trade details?") from an existing 1,102-entry knowledge base — grounded, cited, and never guessed.

Headline capabilities

  • In-chat file delivery, no email round-trip. Reports come back as file cards the client downloads on the spot.
  • Deterministic and auditable. A thin LLM only classifies intent and reads parameters out of free text; hardcoded flows do the fulfilment. The model never improvises a report.
  • A human is always reachable. Every conversation can raise a Freshdesk ticket (with the transcript attached) and shows a call-support chip — which stays visible even after a ticket is raised.
  • Built to improve itself. Every turn is written to a conversation store and traced, so quality can be measured and a fine-tuned model built later.
  • Compliant by construction. A persistent footer reads "Factual answers only — never investment advice", and the bot refuses to give any.

💡 The one-line pitch

Jini turns the most common, most tedious support requests — "send me my report" and "explain this charge" — into a two-tap, self-serve conversation that happens without a human, without an email, and without leaving the app.

3Architecture

How it works

The core design is a deterministic flow engine gated by a thin LLM router. The model decides what the user wants; hardcoded flows decide how to deliver it. This keeps report fulfilment predictable, testable, and safe.

🧭 The LLM router

Classifies each utterance into an intent (a report flow, RAG, or a ticket), pulls out any parameters already present in free text (financial year, dates, segment, format), and asks a follow-up only when genuinely ambiguous. Runs on claude-sonnet-5, with claude-haiku-4-5 as a backend-configurable toggle.

⚙️ The flow engine

Once the intent is known, a hardcoded state machine drives fulfilment — segment / date / FY / format as tappable stepper cards, a deterministic API call, byte validation, and file-card delivery. The LLM never improvises the fulfilment path. Guardrails, prompts, and chips are externalized as config per module.

📌 Model IDs, pinned

The flow spec's "sonnet-4-6" is not a real model ID. The pinned decision (2026-07-16) is claude-sonnet-5 for the router and RAG generation, with claude-haiku-4-5-20251001 as the configurable cheap-model toggle.

The request lifecycle — one turn, end to end

  1. Handoff

    The FinX app opens the chat and passes context as URL query params: userId, sessionId, accessToken (a JWT), isDarkTheme, platform, page. Jini captures both the SessionId and the JWT — different backends need different ones.

  2. Greeting

    The widget shows a time-aware greeting (by Client ID in Phase 1, e.g. "Hey X008593"), four configurable starter chips, and a free-text box. Phase 2 upgrades the greeting to a first name via get-profile.

  3. Route

    The first user action — a chip or a typed message — goes to the LLM router, which classifies the intent and extracts any parameters already in the text.

  4. Collect

    The flow engine drives a deterministic stepper (segment → date range → format, etc.) with tappable chips, asking at most two follow-ups before offering escalation.

  5. Fetch — server-side

    The backend calls the FinX API server-side, fetches the returned report URL, and validates the bytes (a size floor plus magic-byte check). The often-unauthenticated URL never reaches the client or the logs.

  6. Deliver

    The validated bytes are delivered as a chat file card with a friendly, renamed filename. If the bytes fail validation, one silent auto-retry runs; if that fails too, a conversational error bubble with recovery chips appears — never a toast.

  7. Persist & trace

    A separate thread writes the turn to PostgreSQL, and DeepEval traces the turn — decisions and tool calls included, not just the final text.

Caps & escalation

Conversation cap

10 messages per conversation (leaning toward a soft, prompt-side guardrail that tracks a "conversation number"). At the cap, Jini offers to raise a ticket or call support.

Follow-up cap

≤ 2 follow-up questions per ambiguity. If it's still unresolved, Jini stops guessing and offers escalation — a ticket (with the transcript) and the call-support chip.

🧾 A couple of flow-design touches worth knowing

Date pickers hard-disable out-of-range dates instead of validating after the fact — per-flow windows differ by design (Ledger floors at Jan 2019; P&L / Contract Note / Global Detail at Jan 2018; P&L clamps to a 2-year range). Contract notes are keyed by an opaque file_id (never by date), and dual-note days show an NSE·BSE vs MCX segment badge. Brokerage and Holding cards are rendered dynamically from whatever the API returns — segments and rows are never hardcoded.

5Data foundations

Data & knowledge base

Three data assets underpin Jini: the RAG knowledge base (which already exists), the conversation store (which we design), and the trace stream that captures every decision.

The RAG knowledge base — reuse, don't rebuild

What's there

  • Table qa_chunks in PostgreSQL + pgvector.
  • 1,102 rows, embeddings at 3,072 dimensions (text-embedding-3-large, full size).
  • One queryable text column, chunk = Query → Solution → TAT concatenated.
  • Tokens: max 653, mean ~59 — so no sub-chunking needed.

Why sequential scan is correct

pgvector's ANN indexes (HNSW, IVFFlat) cap at 2,000 dimensions; the KB is 3,072-dim. At 1,102 rows a sequential scan is exact and fast, and sidesteps the conflict entirely. To scale later, the option is a Matryoshka truncation to ≤ 2,000 dims. Cosine distance throughout. Query vectors must match the stored model and dimension — so query-time embedding is effectively locked to text-embedding-3-large @ 3072.

⚠️ "Hybrid" still has an open half

The design calls for hybrid retrieval (full-text search + vector), but the FTS half — the tsvector setup and the fusion strategy (e.g. RRF) — plus Top-K and the reranker choice are not yet specified. These are open decisions before implementation, not blockers.

The conversation store — a Phase-1 design task

After every bot response, a separate thread inserts the turn into PostgreSQL. The schema is undefined by design (the owner asked for help deriving it). The natural columns, derived from the trace model:

thread_idturn_iduser_id user_messageassistant_messagedetected_intent tool_calls + args + resultsretrieval_context timestampslatency / tokensmodel_version turn_number

It must capture decisions and tool traces, not just the final text — that's what makes it useful for a future fine-tune.

Tracing & evaluation plumbing

The model

A Trace is one request lifecycle; a Span is one call (agent / retriever / llm / tool); a Thread groups traces that share a thread_id — one conversation. The app owns state; DeepEval only observes.

The tool

DeepEval's @observe instruments every turn and tool call; an optional Confident AI export works fully offline. PII masking is mandatory via a mask= hook — names, emails, Client IDs, and ledger amounts are redacted before anything is exported.

6The hard part

Integration reality

FinX is not one API. Jini's report flows reach across five different hosts with several auth schemes. This is the single biggest engineering reality of the project — one generic client wrapper will not fit; it needs a per-backend adapter set.

🔑 The credential trap: CML fails if you hand it the SessionId

The two middleware backends authenticate with the SessionId. The MIS backend (CML) authenticates with the SSO JWT — the accessToken the frontend hands off. They are different tokens. The widget must capture and forward both, and the backend must route the correct one to each host.

Session ↔ client binding

An Invalid SessionId error usually means a session-to-client mismatch, not an expired token. The .NET backend binds a SessionId to exactly one ClientId: a valid session for X008593 returns 200 for its own client but 401 for any other. The rule is simple — always send the ClientId the widget's session actually owns; never mix. (This is also a good security property: the .NET backend enforces the binding.)

Two error shapes to handle — read the body, then the status

Business failures → HTTP 200

"No data found" and similar come back as HTTP 200 with the outcome inside the body (Status / StatusCode). Branch on the body, never on the HTTP status. Each backend has its own envelope and casing, so two parsers are required.

Auth failures → HTTP 401

Bad or stale credentials return a real HTTP 401 — and the envelope differs per backend. Detect auth failure by the status code; then parse the right shape. Stale and garbage tokens are indistinguishable, so treat any 401 as "re-auth needed".

🧷 Field-level traps, by design

Identity fields differ per endpoint (LoginId="JIFFY" here, UserId="neuron" there, client_id elsewhere); RequestFor is not a global enum (download is 0 on P&L/Ledger but 2 on Tax; email is 1 everywhere); the same from: header is a build tag, not auth; and With_Exp even changes the response shape. These are captured in the API reference and hardcoded per endpoint — deliberately not centralized.

⚠️ The one true blocker: SessionId lifetime

There is no documented login or refresh endpoint anywhere. The widget receives the SessionId from the app handoff, but its acquisition, lifetime, and expiry behaviour are unmeasured. This is the biggest open integration question — everything else is captured.

7Risks we defend against

Security & compliance

Live testing on the FinX infrastructure (2026-07-16) surfaced two real issues. These are upstream platform observations, not Jini bugs — reported so the Jini backend defends against them and the FinX team can be notified. Neither was exploited.

🔴 Finding A — the Contract-Note chain enforces no authentication (IDOR / BOLA)

The contract-note list and download endpoints returned full data with garbage or absent credentials, authorizing purely on the body client_id. The entire chain (list file_ids → download PDFs) is reachable with only a client code.

How Jini defends: it gates every client_id by the widget's authenticated session and never proxies a user-supplied one; it treats file_id values as sensitive (never exposed to the client or logs); and it reports the finding upstream to FinX.

🔴 Finding B — the CML signed URL's expiry is an illusion

The CML link is an S3 pre-signed URL with a 120-second expiry, fronted by CloudFront. Testing showed CloudFront caches by path only — it ignores the signed query string — so once any valid link warms the cache, the same object is served past expiry and on repeat fetches. The "120s / single-use" assumption is false.

How Jini defends: it never relies on URL expiry as a boundary — it fetches server-side within seconds, discards the link, and never surfaces or logs it. (It also notes the CML PDF is a static S3 object, so it may be slightly stale.)

PII handling

get-profile: take one field, drop the rest

The profile endpoint returns heavy PII — PAN, full address, email, date of birth, and a whole bank-account array. Jini extracts only the first name (from FirstHolderName) for the Phase-2 greeting and discards everything else. The full response is never logged, stored, traced, or sent to the client.

Masking & third-party data

Ledger narrations can contain a third party's name/account, and email confirmations leak the full registered address — so emails are always shown masked (san***.harsha@gmail.com) and not editable in-bot. The DeepEval mask= hook redacts names, emails, Client IDs, and amounts before any export.

Compliance guardrails, always on

  • Never investment advice. A persistent footer states it, and RAG answers refuse to give it and cite the KB.
  • Nothing leaks to the user. Report URLs, file_ids, server filenames, and raw API errors are never exposed — errors are conversational bubbles; the raw Reason is logged server-side only. (One deliberate exception: CML keeps its harmless Client_Master_List.pdf name.)
  • Email only to the registered address, masked. Not editable inside the bot — a compliance boundary, not a limitation.
8Quality bar

Evaluation & quality

Quality is measured two ways: automated DeepEval metrics over the RAG and agent behaviour, and a manual test plan of 88 human-authored cases. A single "Blocker" failure holds the launch.

Automated metrics (DeepEval)

Single-turn RAG

The referenceless "RAG triad": Answer Relevancy, Faithfulness (hallucination), and Contextual Relevancy. With labels, add Contextual Precision (ranking) and Recall (embedding), plus a GEval correctness criterion. Each metric diagnoses a specific component — retriever vs generator scored separately.

Multi-turn & agentic

Retrieval happens every turn, so failure modes are context drift and cross-turn hallucination. Headline metric: Conversation Completeness. Also Turn Relevancy, Knowledge Retention, Role Adherence, Tool-Use / Goal Accuracy, and a Conversational GEval that asserts "never gives investment advice".

Test cases are generated by simulation: ConversationalGolden scenarios drive a ConversationSimulator that loops simulated-user → app → outcome check. Returning retrieval_context and tools_called on each turn is required plumbing — it's what unlocks the RAG and tool metrics. Authoring ≥ 20 goldens is an open task; no machine-runnable golden set exists yet.

🚫 The non-negotiable: no cross-client leakage

Cases I1–I3 are all blockers — data must be for the right client, for the exact period, and match the backend figures. G1/G5 verify the AuthToken only ever returns the session's own client. A failure here holds the launch, full stop.

🎯 Thresholds are still a decision

No numeric pass/fail thresholds are mandated anywhere — the tutorial defaults (0.7 / 0.8) are illustrations, not mandates. Picking per-metric thresholds and the DeepEval judge model is an explicit open decision. (Note: the spreadsheet README counts, 36/46, don't match the actual sheet counts, 41/47 — reconcile.)

9Where we are

Status & what's next

Discovery and spec are essentially complete. Every report flow's endpoints — data schemas, file-delivery legs, auth-failure envelopes, session binding, and the Freshdesk structure — have been live-tested and captured. The remaining items are confirmations and decisions, not blockers.

✅ Captured

  • All report data schemas (Ledger, P&L, Detailed P&L).
  • All file-delivery endpoints (PNL/Ledger/Tax PDF+Excel, contract-note list + download, CML).
  • Auth-failure envelopes per backend; session ↔ client binding rule.
  • get-profile (first-name-only), Brokerage, Holdings; Freshdesk account structure & field cascade.

⚠️ Confirmations owed

  • MTF Margin discriminator — Margin:0/:1 were byte-identical on a no-MTF account; needs one capture from an account with live MTF holdings.
  • Holdings FINX-JWT provenance — where the backend obtains the FINX-issued JWT (distinct from the SSO JWT) is unresolved.
  • Holding & Global-Detail file delivery — data endpoints exist, but no PDF leg; decide data-card vs capture a file endpoint.
  • SessionId lifetime — acquisition and expiry still unmeasured.

🧱 Blocked, not stubbed

Flows without a captured file endpoint wait rather than ship as stubs. The launch proposals cover only flows with known endpoints: P&L, Ledger/MTF, Contract Note (list), Tax/CG, and CML — plus RAG, tracing, the conversation store, and the widget shell.

The build plan: contracts first

Per the project's parallel-workflow rules, a contracts-first change must land in main before any parallel fan-out. It defines the shared interfaces every downstream task depends on, so tasks never fight over the same files:

1 · Widget ↔ backend wire contract

The headline gap — chat endpoint schema, streaming decision, and render-block types (bubble, chip row, stepper, file card, note-list, error bubble).

2 · Router I/O contract

The Intent enum, ExtractedParams, and RouterResult.

3 · FinXClient interface

The per-backend adapter set, three envelope parsers, typed models, and capture fixtures.

4 · Flow state machine

The Flow/Step machine plus the guardrail config schema.

5 · Conversation-store migration

The DB migration — a single-owner task.

6 · Remote-config schema

What's New, limits, chips, greeting pool, product list.

7 · Tracing conventions

Span taxonomy, configure() setup, and the mask signature.

8 · Error taxonomy

The shared E-NODATA / E-YEAR / E-TIMEOUT / E-FETCH / E-UNKNOWN config.

9 · LLM client wrapper

Pinned model IDs plus the Haiku toggle.

⚠️ Merge-conflict watch

The Intent enum, the FinXClient, and the remote-config schema are files every flow task will want to touch. All intents, methods, and config keys are defined up front, and flow tasks are forbidden from editing them.
