# Tasks: widget-shell

Ordered, verifiable tasks derived from `proposal.md` + the frozen contract
`app/contracts/schema/chat_wire.schema.json`. Tasks never expand scope beyond
the proposal. Each has a `Done:` criterion checkable by command or test.

Contract-resolution note (both proposal [CONFIRM]s are resolved by the frozen
schema, not reinterpreted):
- Wire types are **generated from** `app/contracts/schema/chat_wire.schema.json`
  (the emitted JSON-schema artifact the proposal asked about) — `wireTypes.ts`
  is generated + committed, not hand-mirrored.
- Remote-config is **folded into the first `/api/chat` response** as
  `config_slice` (present in `ChatResponse`), so the widget's sole network
  surface is `POST /api/chat`. No separate remote-config GET.
- The frozen schema is authoritative where it differs from the illustrative
  prose table in proposal.md: block `type` discriminators are underscore-cased
  (`chip_row`, `stepper_card`, `note_list_card`, `error_bubble`,
  `ticket_confirmation`, `data_card`, `file_card`); bot and user bubbles are
  separate types (`bubble`, `user_bubble`); `generating` is a first-class block.

Own dependency graph (widget/package.json + lockfile — the only frontend deps in
the repo). Vite + React 18 + TS, vitest + jsdom + @testing-library/react, MSW,
json-schema-to-typescript, ajv (fixture validation), webmcp-types (devDep).
Scripts: `dev`, `dev:mock`, `build`, `typecheck` (tsc --noEmit), `test` (vitest
run), `gen:types` (schema → wireTypes).
Done: `npm --prefix widget install` succeeds; `npm --prefix widget run typecheck`
exits 0; `npm --prefix widget test` runs a passing smoke test.

## T4 — Bootstrap from URL params
`src/bootstrap.ts` parses the six params (`userId`, `sessionId`, `accessToken`,
`isDarkTheme`, `platform`, `page`) once → `SessionContext` (with `entry_surface`
derived from `page`). Held in memory; `accessToken` never persisted/logged.
Done: unit tests for param parsing, `entry_surface` derivation (support/reports),
and that `accessToken` is absent from localStorage and from any console output.

## T5 — Mock server + fixtures
`mock/server.ts` (MSW handler implementing `POST /api/chat` per the wire
contract: config_slice only on first turn, thread_id/turn_number threading,

(support/reports config_slice) + a stepper→file-card flow + a note-list page.
Done: an ajv test validates every fixture against the frozen schema; a test
asserts config_slice present on turn 0 only.

## T6 — chatClient + conversation state
`src/api/chatClient.ts` (`POST /api/chat`; transport/HTTP failure → synthesized

`src/state/conversation.ts` (append-only block log; one POST per user action;
turn_number increments; thread_id threaded; caps surfaced).
Done: tests assert exactly one POST per action, blocks appended in order,
transport failure yields an `E-TIMEOUT` error_bubble, accessToken echoed in
request but never logged.

## T7 — Render-block component set
One component per wire type under `src/blocks/` + `RenderBlock.tsx`
discriminated-union switch on `block.type`: BubbleBlock (bubble + user_bubble),

appends server blocks in order and never synthesizes/reorders.
Done: each block renders from its fixture and matches its prototype screen
counterpart; RenderBlock switches on type; unknown type is a safe no-op.

## T8 — Interaction contracts
